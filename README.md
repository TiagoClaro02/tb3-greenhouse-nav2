# TurtleBot3 Greenhouse Row-Following Mission

Boustrophedon row-following mission for a simulated TurtleBot3 in a greenhouse,
built on ROS2 Jazzy, Gazebo Harmonic, and Nav2.

## 1. Quick Start

```bash
# Dependencies (Ubuntu 24.04, ROS2 Jazzy)
sudo apt install ros-jazzy-navigation2 ros-jazzy-nav2-bringup \
    ros-jazzy-nav2-simple-commander ros-jazzy-turtlebot3-msgs \
    ros-jazzy-dynamixel-sdk ros-jazzy-hls-lfcd-lds-driver

mkdir -p ~/turtlebot3_ws/src && cd ~/turtlebot3_ws/src
git clone -b jazzy https://github.com/ROBOTIS-GIT/turtlebot3.git
git clone -b jazzy https://github.com/ROBOTIS-GIT/turtlebot3_simulations.git
cd ~/turtlebot3_ws && rosdep install --from-paths src --ignore-src -r -y
colcon build --symlink-install

# Clone and compile Repo
git clone <this-repo-url> ~/greenhouse_mission
cd ~/greenhouse_mission
colcon build --symlink-install

source /opt/ros/jazzy/setup.bash
source ~/turtlebot3_ws/install/setup.bash
source ~/greenhouse_mission/install/setup.bash

export TURTLEBOT3_MODEL=waffle
export GZ_IP=127.0.0.1

# Launch of Gazebo Environment + Nav2 + TurtleBot 3
ros2 launch greenhouse_mission greenhouse_bringup.launch.py

# Mission Start
ros2 run greenhouse_mission mission_node
```

## 3. Simulation Setup

**World generation is parametric, not hand-authored.** `geometry_config.py` defines
`NUM_CORRIDORS`, `ROW_WIDTH`, `ROW_LENGTH`, `HEADLAND`; `generate_world.py`,
`generate_map.py`, and `generate_waypoints.py` derive the SDF world, the static
map, and the mission waypoints from it. Changing a dimension means editing one
file and re-running three scripts — no hand-edited SDF/YAML to keep in sync.
This was a deliberate choice given the stated follow-up session changes a
dimension live.

## 4. Row-Following Mission

`mission_node.py` uses `nav2_simple_commander`'s `BasicNavigator`. For each row:
drive to the entry waypoint, then drive to the exit waypoint (monitored — see
Section 6). Progress and outcomes are published as plain strings on
`/mission_status`.

**Row direction is decided at runtime.** A naive
boustrophedon alternates entry side (west/east/west...) by row index — but
that breaks the moment a row is skipped: the robot never reaches the far side
of a blocked row, so forcing the next row's entry to the "expected" side sends
it on an unnecessary crossing back to a side it never reached. `current_side`
is tracked as real state: it flips only when a row is genuinely completed, and
stays put when a row is aborted.

## 5. Tuning of Nav2 Params

| Parameter | Default | Changed to | Why |
|---|---|---|---|
| `inflation_radius` (both costmaps) | 0.70 | **0.78** | Per Nav2's own tuning guide: cost-aware planners (NavFn, Theta*, Smac) produce suboptimal paths in flat zero-cost regions; a smooth potential across the whole navigable width — even exceeding half the row width — lets the planner's inherent per-cell distance cost correctly discourage long detours, rather than being confused by flat cost near a partial obstacle. |
| `cost_scaling_factor` (both costmaps) | 3.0 | **8.0** | Paired with the larger radius above; an increase on this values allows the cost function to output solutions closer to the obstacles, in the shown scenarios the default value or even a smaller on would result in a safer, and more optimal path, however with this param the robot is capable of crossing narrow paths, and avoid obstacles that only cover part of the width of the row. |
| `enable_stamped_cmd_vel` (controller_server, velocity_smoother, collision_monitor, behavior_server, docking_server) | `false` | **`true`**, all five | Jazzy compatibility requirement |
| `amcl.update_min_d` / `update_min_a` | 0.25 / 0.2 | *0.15 / 0.1* | Long, feature-sparse rows leave AMCL's along-corridor position weakly constrained by scan matching, relying mostly on odometry between corrections; larger thresholds meant each correction reconciled more accumulated drift in one visible jump. More frequent, smaller corrections reduced a low-frequency oscillation during straight-row driving. |

## 6. Recovery Behavior

Every row's headland connects to every other row's headland at both ends — this is required by the
spec, and correctly modeled in the world. That means Nav2's global planner can
"solve" a blocked row by detouring through a neighboring row and back in,
reaching the correct final waypoint position without ever driving the blocked
row. `goToPose` would report `SUCCEEDED` even though the row was never
traversed.

The
robot has no way to know about a blockage until its own sensors observe it, so a `getPath()`
check made before departure only sees the static map and would never catch
this. Instead, `mission_node.py` subscribes to `/plan` and compares its length against
`max_path_length_m` continuously, for the whole duration of each leg. As soon
as a replan exceeds the threshold — exactly when the obstacle is sensed and
the planner reroutes via a neighboring row — the task is cancelled
immediately, rather than letting the robot drive the full detour.

The threshold is geometry-derived:
`ROW_WIDTH + hypot(ROW_WIDTH, ROW_LENGTH)`, computed once in
`generate_waypoints.py` and read from `waypoints.yaml` at runtime. This formula considers the maximum
lenght the robot must do per row, accounting with the possibility that is could need to avoid some obstacles within the row.
See Section 7 for a test that
demonstrates this formula's own limits.

**Mission-node level, not a custom BT plugin.** Chosen for testability (the
detection/reordering logic is plain Python, unit-testable with no ROS
dependency — Section 7) and debuggability under a time-box. The trade-off:
a BT-native implementation would tick reactively as part of Nav2's own
execution model and be visualizable/reusable; the mission node
instead polls in a loop, which is functionally equivalent but less "native"
to Nav2's architecture.

## 7. Testing

Two test files, both plain `pytest`, no Gazebo/Nav2 required to run:

**`mission_test.py`** — the pure decision logic (`path_length`,
`compute_leg`, `next_side`, `compute_max_path_length`), extracted into
`mission_logic.py` specifically so it has no ROS import. 15 tests; `test_safety_property_BREAKS_for_wide_short_rows`
proves the detour-detection formula is not a universally valid geometric
bound — it's correct for this project's actual proportions (narrow, long
rows) but breaks for `ROW_WIDTH ≳ 4.0` at this project's `ROW_LENGTH`/
`HEADLAND`. Left failing and documented rather than "fixed," since a general
formula would need real path-planning geometry, not a triangle-inequality
approximation, and this project's rows are narrow by definition.


## 8. Known Limitations

- **AMCL drift in long, feature-sparse corridors.** A well-known limitation
  of particle-filter localization: along-corridor position is weakly
  constrained by scan matching in a long straight row, relying mostly on
  odometry between corrections. Mitigated (Section 5), not eliminated.
- **`max_path_length_m`'s safety margin is geometry-dependent** (Section 7)
  — valid for this project's actual dimensions, not a universal bound.
- **Manual-sync couplings**, not yet fully automated: AMCL's
  `initial_pose` in `nav2_params.yaml` must match `waypoints.yaml`'s
  `spawn_x`/`spawn_y` by hand after a geometry change (the launch file's
  spawn pose reads this automatically; `nav2_params.yaml` does not).
- **`smoother_server` is unused** — the default Nav2 behavior
  tree never invokes it. A custom BT wiring it in would output smoother 
  paths with less oscilations.
- Controller smoothness has room for further tuning. Linear and angular
  velocity still show some residual oscillation that a longer tuning pass
  would likely improve further. Wiring the existing but currently-unused
  smoother_server into the behavior tree is the most likely next step,
  but wasn't pursued to completion in this time-box.
